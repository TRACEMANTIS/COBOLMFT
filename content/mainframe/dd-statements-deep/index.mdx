# DD statements deep

A DD (Data Definition) statement names a file the program will use,
points it at a dataset, and tells the system what to do with the
dataset before, during, and after the step. Most JCL bugs are DD bugs:
wrong DSN, wrong DISP, wrong RECFM. This lesson is the parameter
reference and the set of patterns that account for nearly everything
you'll see in production.

## The shape of a DD

```jcl
//CUSTFILE DD  DSN=PROD.CUST.MASTER,
//             DISP=SHR,
//             RECFM=FB,
//             LRECL=120,
//             BLKSIZE=0
```

- `//CUSTFILE` — the **DDname**, matched to the program's `ASSIGN`.
- `DD` — the card type.
- The rest — parameters, comma-separated, continued onto multiple
  cards by ending each line with a comma.

The DDname is what the COBOL program references. The DSN is what z/OS
references in the catalog.

## DSN — the dataset name

`DSN=` (or `DSNAME=`) is the dataset's catalog name. Conventional
shops use a hierarchical naming scheme:

```
PROD.CUST.MASTER             ← production master
PROD.CUST.MASTER.G0042V00    ← GDG generation
SYS1.PROCLIB                 ← system PROC library
USER01.TEST.JCL              ← user dataset
&&TEMPWORK                   ← temporary (job-local)
```

Conventions vary, but you'll see the high-level qualifier (HLQ) used
to designate ownership/environment (`PROD`, `TEST`, `SYS1`, individual
user IDs). Shops have firm rules about which HLQs are production,
which are dev, which need RACF authority. **Look at the HLQ first** —
it tells you the dataset's role.

### Special DSN forms

- `DSN=&&temp` — temporary, job-local dataset.
- `DSN=*.STEP1.OUTFILE` — back-reference to another step's DD.
- `DSN=PROD.MYGDG(0)` — current GDG generation.
- `DSN=PROD.MYGDG(+1)` — next (new) GDG generation.
- `DSN=PROD.MYGDG(-1)` — previous (older) generation.

GDG (Generation Data Group) syntax is its own topic; the brief version
is that a GDG is a series of dataset generations sharing a base name,
and the relative number selects which one.

## DISP — disposition

`DISP=(status,normal-end,abnormal-end)`. Three positions:

### Status (first position)

| Value | Meaning                                                        |
| ----- | -------------------------------------------------------------- |
| `NEW` | Create the dataset. SPACE parameter required.                   |
| `OLD` | Exclusive access to an existing dataset. Other jobs wait.       |
| `SHR` | Shared access to an existing dataset. Multiple readers OK.      |
| `MOD` | OLD if the dataset exists, append to end; NEW if not.           |

### Normal-end (second position)

| Value     | Meaning                                                   |
| --------- | --------------------------------------------------------- |
| `KEEP`    | Leave the dataset on disk; do not catalog if not already. |
| `DELETE`  | Erase the dataset.                                         |
| `PASS`    | Pass to a later step in the same job (job-local).          |
| `CATLG`   | Catalog the dataset (newly created datasets are usually). |
| `UNCATLG` | Remove from catalog but leave on disk.                    |

### Abnormal-end (third position)

Same values as normal-end. Fires only if the step ABENDs.

### Common patterns

```jcl
DISP=SHR                   *> read-only existing dataset
DISP=OLD                   *> read/write existing, exclusive
DISP=(NEW,CATLG,DELETE)    *> create, catalog on success, clean up on abend
DISP=(NEW,PASS)            *> create, pass to next step (temp use)
DISP=(MOD,KEEP)            *> append to existing log file, keep
```

`DISP=SHR` for read-only is the most common; **using `OLD` for a
read-only step is a real bug** because it locks the dataset against
concurrent jobs unnecessarily. Watch for this in legacy JCL.

If only the first position is given (`DISP=SHR`), the second defaults
to `KEEP` for existing datasets and `DELETE` for new.

## SPACE — allocating new datasets

For `NEW` datasets, you must declare the space:

```jcl
SPACE=(CYL,(50,10),RLSE)
```

- `CYL` — units. Alternatives: `TRK` (tracks), `BLK` (blocks).
- `(50,10)` — primary, secondary. Initial 50 cylinders, then 10 more
  when needed (up to 15 secondary extensions on most DASD).
- `RLSE` — release unused space at close.

Common SPACE patterns:

```jcl
SPACE=(CYL,(10,5),RLSE)             *> small dataset, ~10 cyl
SPACE=(CYL,(100,50),RLSE)           *> medium
SPACE=(CYL,(500,250),RLSE)          *> large
SPACE=(TRK,(15,5,10),RLSE)          *> partitioned (PDS): 10 directory blocks
```

For PDS/PDSE (libraries with members) the third element of the inner
list is the directory block count. PDSE is the modern replacement —
similar SPACE shape, more reliable.

The maintainer's instinct: when a job ABENDs with `B37` or `D37`
(out-of-space), the SPACE parameter is too small for the data the job
is producing. Increase primary, or increase secondary, or both.

## RECFM, LRECL, BLKSIZE — record format

These three parameters describe the dataset's physical layout:

### RECFM — record format

| Value  | Meaning                                                        |
| ------ | -------------------------------------------------------------- |
| `F`    | Fixed-length, unblocked (one logical record per physical block) |
| `FB`   | Fixed-length, blocked (multiple records per block)              |
| `V`    | Variable-length, unblocked                                       |
| `VB`   | Variable-length, blocked                                         |
| `U`    | Undefined (catch-all; uncommon outside utility datasets)         |
| `FBA`  | Fixed-block with ANSI carriage control                           |
| `VBA`  | Variable-block with ANSI carriage control                        |

For COBOL data files: nearly always `FB` (fixed records, sequential)
or `VB` (records of varying length, common with `OCCURS DEPENDING
ON`). Match the program's record description to the file's RECFM —
mismatch produces FILE STATUS 04 (record length wrong) or 39 (RECFM
incompatible).

### LRECL — logical record length

The program's record length, in bytes:

```jcl
RECFM=FB,LRECL=120     *> 120-byte records
RECFM=VB,LRECL=2004    *> max 2000 bytes plus 4-byte length prefix
```

For RECFM=V/VB, the LRECL is the **maximum** record length, including
the 4-byte block descriptor word.

### BLKSIZE — physical block size

The unit of physical I/O. Larger blocks mean fewer I/Os and faster
sequential reads, up to a sensible limit:

```jcl
RECFM=FB,LRECL=120,BLKSIZE=0          *> system-determined (recommended)
RECFM=FB,LRECL=120,BLKSIZE=27840      *> classic half-track on 3390 (232 records)
```

`BLKSIZE=0` lets the system pick an optimal value based on the device
(half-track or full-track). **Modern shops nearly always use 0.**
Hardcoded values appear in legacy JCL as historical artifact.

## UNIT and VOL — physical placement

`UNIT` declares the device class:

```jcl
UNIT=SYSDA              *> any DASD pool the installation has named SYSDA
UNIT=TAPE               *> tape drive
UNIT=3390               *> specific device type
```

`VOL=SER=volser` pins to a specific volume:

```jcl
DSN=USER01.PRIVATE.JCL,VOL=SER=PRV001,UNIT=3390
```

Most production JCL uses neither — the catalog already knows where the
dataset lives, and SMS (system-managed storage) places new datasets
automatically. You'll see UNIT/VOL in older code and in special-purpose
allocations.

## SYSOUT — print classes

```jcl
//SYSPRINT DD  SYSOUT=*
//REPORT   DD  SYSOUT=A
```

`SYSOUT=*` routes the output to the same class the job is using (set
by `MSGCLASS=` on the JOB card). Specific classes (A, X, Z, etc.) route
to specific spool destinations; the conventions are shop-dependent.

`SYSOUT=*` for `SYSPRINT` (compile or sort messages) and `SYSPRINT`
for runtime DISPLAYs is the most common pattern — they end up in the
job's spool, viewable in SDSF.

## DUMMY — null I/O

```jcl
//OPTLOG DD  DUMMY
```

The program can `OPEN`, `READ`, `WRITE` and so on, but:

- `READ` returns AT END immediately (the file is empty).
- `WRITE` is a no-op.

Use cases:

- A program is shared between callers that need to write a log and
  callers that don't. Pointing the optional log DD at `DUMMY` skips
  it.
- Test runs that want to suppress real output.
- Debugging — point the output DD at DUMMY to see if the rest of the
  program runs without I/O.

## A worked DD: the customer master

```jcl
//CUSTMSTR DD  DSN=PROD.CUST.MASTER,
//             DISP=SHR
//
//CUSTNEW  DD  DSN=PROD.CUST.MASTER.NEW(+1),
//             DISP=(NEW,CATLG,DELETE),
//             SPACE=(CYL,(500,250),RLSE),
//             RECFM=FB,
//             LRECL=120,
//             BLKSIZE=0
//
//CUSTBAD  DD  DSN=&&BADCUST,
//             DISP=(NEW,PASS),
//             SPACE=(CYL,(10,5),RLSE),
//             RECFM=FB,
//             LRECL=120,
//             BLKSIZE=0
//
//SYSPRINT DD  SYSOUT=*
//SYSUDUMP DD  SYSOUT=*
//CEEDUMP  DD  SYSOUT=*
```

Read it:

- `CUSTMSTR`: existing production master; shared (read-only) access.
- `CUSTNEW`: new GDG generation `(+1)`; created, cataloged on success,
  cleaned up on ABEND. 500 cylinders primary, 250 secondary. RECFM=FB,
  120-byte records, system-chosen blocksize.
- `CUSTBAD`: temporary error-record file (`&&BADCUST`); passed to a
  later step for analysis; gone at job end.
- `SYSPRINT/SYSUDUMP/CEEDUMP`: DISPLAY output and dump output go to
  the spool.

This pattern — read-shared input, new-cataloged-with-cleanup output,
temp scratchpad, dump destinations — is what 90% of batch JCL looks
like. Memorise it.

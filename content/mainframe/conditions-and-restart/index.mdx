# Conditions and restart

A real production batch job has steps that depend on each other. Step
1 reads a master file, step 2 produces an extract, step 3 sorts the
extract, step 4 loads the result. If step 2 fails, you do not want
step 3 to run on garbage. If step 4 fails after step 3 succeeded, you
want to restart from step 4 without re-running everything.

This lesson is about the JCL primitives for conditional execution
(`COND`, `IF/THEN/ELSE/ENDIF`) and the design discipline that makes a
job restartable. By the end you should be able to read a multi-step
job, identify its restart points, and choose where to kick off a
recovery run.

## Return codes

Every step ends with a **return code** — a number set by the program
or, on ABEND, by the system:

| RC range  | Convention                                                |
| --------- | --------------------------------------------------------- |
| `0`       | Clean success                                              |
| `4`       | Warning — process completed but with notable issues       |
| `8`       | Errors — partial completion                                |
| `12`      | Severe errors                                              |
| `16+`     | Fatal — the program decided to terminate                   |
| `4095`    | (rare) maximum RC value                                    |

For COBOL programs, the return code is the contents of the special
register `RETURN-CODE` at end-of-program:

```cobol
*  Set RC explicitly before STOP RUN
MOVE 8 TO RETURN-CODE.
STOP RUN.
```

Default is 0. ABENDs do not set a return code; the system code (e.g.,
`S0C7`) is reported separately, and downstream COND/IF tests treat
ABENDed steps differently from RC-failures.

The shop-wide convention is the load-bearing piece: if a payroll
extract returns 4 to mean "no records found, but proceed" and the
downstream report uses `COND=(4,LT)` (run if RC <= 4), the program
flows. Different shops have different conventions; what's "warning"
in one is "error" in another.

## COND — the legacy conditional syntax

`COND` on an EXEC card tests prior steps' return codes:

```jcl
//STEP3   EXEC PGM=ABC,COND=(4,LT,STEP1)
```

Read this as: **skip STEP3 if `4 LT STEP1`'s RC**. Skip if step 1's
RC is greater than 4. Run if step 1's RC is 4 or less.

The full COND syntax is:

```
COND=(value,operator[,stepname])
```

- **value** — constant on the left side of the comparison.
- **operator** — `EQ`, `NE`, `LT`, `LE`, `GT`, `GE`.
- **stepname** — optional; if omitted, applies to all preceding steps.

The mental model people start with — "this is the condition under which
the step runs" — is **wrong**. The COND test is the condition under
which the step is **skipped**. This inverted logic is a permanent
JCL stumbling block; if you find it confusing, you are not alone.

### Multiple conditions

```jcl
//STEP3   EXEC PGM=ABC,COND=((4,LT,STEP1),(0,NE,STEP2))
```

Multiple `(value,op,stepname)` tuples. **STEP3 is skipped if ANY of
the tests are TRUE** (logical OR of skip conditions).

### COND on the JOB card

```jcl
//MYJOB JOB CLASS=A,COND=(8,LT)
```

Job-level COND tests every step. If `8 LT (any step's RC)`, the
remaining steps are skipped. Used to terminate the job early on
serious errors.

### COND=ONLY and COND=EVEN

```jcl
//STEP9   EXEC PGM=CLEANUP,COND=ONLY
```

`COND=ONLY` means "run this step ONLY if a previous step ABENDed."
Used for cleanup or notification steps that should run **only** on
failure.

```jcl
//STEP9   EXEC PGM=CLEANUP,COND=EVEN
```

`COND=EVEN` means "run this step EVEN if a previous step ABENDed."
Used for cleanup steps that must always run.

## IF/THEN/ELSE/ENDIF — the modern syntax

JCL gained `IF/THEN/ELSE/ENDIF` in the late 1980s and most modern
shops use it for new code:

```jcl
//STEP1   EXEC PGM=EXTRACT
//STEP2   EXEC PGM=SORT
//*
//IF (STEP1.RC = 0 AND STEP2.RC = 0) THEN
//STEP3   EXEC PGM=LOAD
//ENDIF
//*
//IF (STEP1.RC > 0 OR STEP2.RC > 0) THEN
//ALERT   EXEC PGM=PAGE,COND=EVEN
//ENDIF
//*
//IF (ABEND) THEN
//CLEAN   EXEC PGM=CLEANUP
//ENDIF
```

What this gets right:

- Reads forward — `IF (cond) THEN run X` instead of `if (skip-cond)
  skip X`.
- Composable boolean logic with `AND`, `OR`, parentheses.
- `ABEND` and `NOT ABEND` test for any prior abnormal termination.

A real shop's JCL is a mix: legacy steps use COND; recent code uses
IF/THEN/ELSE/ENDIF. As a maintainer you must read both fluently.

### Symbolic step references

```jcl
IF (STEP1.RC > 0) THEN ...
IF (STEP1.STEP2.RC > 0) THEN ...     *> step inside an EXEC PROC
IF (PAYROLL.PAYBL010.RC > 0) THEN ... *> if PAYROLL is a PROC name
```

Inside a PROC, the step names you reference are the PROC's internal
step names. Outside (in the invoking JCL), you may need to qualify
with the EXEC step name that invoked the PROC.

## RESTART — kicking off mid-job

When a multi-step job fails partway, you can restart it from any step
without re-running the prior ones:

```jcl
//MYJOB JOB CLASS=A,RESTART=STEP030
```

Or for a step inside a PROC:

```jcl
//MYJOB JOB CLASS=A,RESTART=PAYROLL.PAYBL020
```

The job runs as if every step before `STEP030` (or `PAYROLL.PAYBL020`)
had completed with RC=0. COND and IF tests against those skipped
steps see "RC=0," not "did not run."

This works **only if the prior steps' outputs are still in the right
state**. If the failed step needs the previous step's output dataset,
that dataset must still exist and be correct. Designing for this
condition is the point of the rest of the lesson.

## GDG considerations on restart

GDGs are the most common restart hazard. Recall:

- `DSN=PROD.MYGDG(+1)` references the next (new) generation.
- A successful step with `DISP=(NEW,CATLG,DELETE)` on `(+1)`
  **commits** that generation — it becomes the new (0).
- Subsequent steps in the same job that reference `(+1)` continue to
  see "the next generation" as the same generation just created.
- After job end, the GDG numbering stabilises; what was `(+1)` is now
  `(0)`.

The hazard:

```jcl
//STEP01  EXEC PGM=ABC
//OUT     DD   DSN=PROD.MYGDG(+1),DISP=(NEW,CATLG,DELETE)
//*
//STEP02  EXEC PGM=DEF
//OUT     DD   DSN=PROD.MYGDG(+2),DISP=(NEW,CATLG,DELETE)
//*
//STEP03  EXEC PGM=GHI    ← ABENDs here
//OUT     DD   DSN=PROD.MYGDG(+3),DISP=(NEW,CATLG,DELETE)
```

After the ABEND in STEP03:

- `(+1)` (now `(0)`) is committed — STEP01 succeeded.
- `(+2)` (now `(-1)` from the just-finished position? No — actually
  `(+2)` becomes `(0)` from STEP02's perspective; for the still-active
  job, the relative numbers persist; once the job is over, the
  catalogue commits the new generations).

Actually: GDG numbering is **stabilised at job end**, not at each
step. So during the job, `(+1)`, `(+2)`, `(+3)` are placeholders. At
job end, they become real catalogued generations.

But here's the catch: when an ABEND happens mid-job, **the
already-committed steps' DISP=(NEW,CATLG,DELETE) on `(+N)` typically
defaults to KEEP-on-abend** unless explicitly DELETE. Modern shops
configure this carefully; older configurations leave generations
in inconsistent states.

If you `RESTART=STEP03` after fixing the bug, STEP03's
`DSN=PROD.MYGDG(+3)` references "the third new one" — but is that
the third counting from before the original run, or counting from
now? The answer depends on whether the previous run's generations
were left in the catalog or not, and **this is one of the most
common sources of mysterious "duplicate dataset" errors on restart**.

The defensive fix:

- At restart prep, run an `IDCAMS DELETE` step to clean up any
  partially-committed GDG generations from the failed run.
- Or design steps so they tolerate "the dataset already exists" —
  delete-and-recreate inside the step.
- Or use absolute generation names (`PROD.MYGDG.G0042V00`) instead of
  relative `(+N)`, so the names are unambiguous regardless of
  catalog state.

## Designing restartable jobs

Three design rules turn an arbitrary multi-step job into a
restartable one:

### 1. Each step's output is created fresh

```jcl
//STEP02  EXEC PGM=ABC
//OUTFILE DD   DSN=PROD.STEP02.OUTPUT,
//             DISP=(NEW,CATLG,DELETE),
//             ...
```

`(NEW,CATLG,DELETE)` means: new at start, catalog on success, delete
on ABEND. Re-running this step from scratch always starts with a
fresh dataset. The previous dataset (if any) needs to be cleaned up
first — typically by an `IDCAMS DELETE` before the EXEC, or by
configuring the dataset's DCB to allow `DISP=(MOD,...)` semantics.

### 2. Side effects are idempotent

If a step writes to a database, a queue, or an external system,
re-running it must not duplicate or corrupt:

- Use a **transaction ID** — store one in a control table at start
  of step, check it on re-entry, skip the side effect if already
  recorded.
- Use **upsert** semantics — `MERGE` or `INSERT...ON DUPLICATE KEY
  UPDATE` instead of `INSERT`.
- Wrap multi-row writes in **commit boundaries** that align with
  re-runnable units.

### 3. Steps fail clearly

If a step partially-succeeds (some records written, some not), the
step must either:

- Roll back to a clean state on failure, OR
- Record exactly which records were processed so a restart can resume
  from the right point.

Programs that say "wrote 47 records, then failed at row 48" without
specifying which 47 are unrestartable in any safe way.

## A worked job

```jcl
//PAYROLL JOB CLASS=A,MSGCLASS=X
//*
//STEP010 EXEC PGM=PAYBL010
//CUSTMSTR DD  DSN=PROD.CUST.MASTER,DISP=SHR
//OUTFILE DD   DSN=PROD.PAYROLL.EXTRACT(+1),
//             DISP=(NEW,CATLG,DELETE),
//             SPACE=(CYL,(50,10),RLSE),
//             RECFM=FB,LRECL=120,BLKSIZE=0
//*
//STEP020 EXEC PGM=SORT,COND=(0,LT,STEP010)
//SORTIN  DD   DSN=*.STEP010.OUTFILE,DISP=OLD
//SORTOUT DD   DSN=PROD.PAYROLL.SORTED(+1),
//             DISP=(NEW,CATLG,DELETE),
//             SPACE=(CYL,(50,10),RLSE),
//             RECFM=FB,LRECL=120,BLKSIZE=0
//SYSIN   DD   *
  SORT FIELDS=(1,8,CH,A)
/*
//*
//STEP030 EXEC PGM=PAYBL020,COND=(0,LT,STEP020)
//PAYIN   DD   DSN=*.STEP020.SORTOUT,DISP=OLD
//PAYOUT  DD   DSN=PROD.PAYROLL.PAYMENTS(+1),
//             DISP=(NEW,CATLG,DELETE),
//             SPACE=(CYL,(50,10),RLSE),
//             RECFM=FB,LRECL=120,BLKSIZE=0
//*
//IF (STEP030.RC > 0 OR ABEND) THEN
//ALERT   EXEC PGM=PAGEOPS
//ENDIF
```

Reading this:

- 3 main steps + 1 conditional alert.
- Each main step uses `(NEW,CATLG,DELETE)` and creates a fresh GDG
  generation. Re-running with `RESTART=STEP030` requires the
  intermediate datasets to exist and be valid (they were created in
  steps 010 and 020 of the original run).
- COND on STEP020 and STEP030 mean: "skip if any prior step had RC > 0."
  So failure cascades cleanly.
- The IF block runs the operations alert if STEP030 failed or ABENDed.
- The job is restartable from STEP030 if STEP020's output is still
  valid; from STEP020 if STEP010's is.

The restart procedure for a STEP030 failure:

1. Diagnose and fix the cause (typically a code or data issue).
2. If the GDG generations from STEP010 and STEP020 are valid, the
   restart is just `RESTART=STEP030`.
3. If they are corrupt or partial, you may need to IDCAMS DELETE the
   bad generations first, then restart from STEP010 or STEP020.

The maintainer's instinct: when triaging a failed batch, **look at
the GDG state** before deciding where to restart. A clean GDG state
is the precondition for an easy restart.

# Kernel bypass

The default path of a packet through Linux:

1. NIC raises an interrupt.
2. Top-half handler schedules a softirq.
3. Softirq pulls the descriptor, allocates an `sk_buff`, runs the
   protocol stack, queues to the socket.
4. Application wakes, calls `recv()`, syscall, copy from kernel to
   user space.

That's micro­seconds and a lot of context switches. **Kernel bypass**
moves the NIC's ring buffers into user space and lets the application
poll them directly — no syscall, no interrupt, no copy.

Three flavours you'll meet:

- **Onload (Solarflare/AMD).** `LD_PRELOAD`-style — overrides Berkeley
  sockets. Existing applications get bypass with no source changes.
- **DPDK (Intel-led, vendor-neutral).** A user-space framework where
  the application owns the NIC, drives ring buffers, and implements
  its own protocol stack. Maximum control, maximum porting effort.
- **RDMA (InfiniBand / RoCE / iWARP).** The NIC reads and writes the
  remote application's memory directly. Different programming model
  (verbs, queue pairs) but the lowest latencies for compatible
  workloads.

<Callout kind="tip" title="When NOT to use bypass">
Bypass is strictly worse if the workload doesn't need the latency.
You burn a CPU core polling, you trade kernel correctness for
application correctness, and you lose anything the kernel was doing
for you (firewall, packet capture, multi-tenancy). Reach for it when
you have measured a real need.
</Callout>

# String handling

COBOL was built to push fixed-width records around. Anything more
flexible — building a comma-delimited output, parsing a CSV header,
trimming a trailing-space-padded name — needs the four string verbs.
They are not glamorous and they are easy to use wrong; this lesson
covers the patterns that work and the ones that quietly do not.

## Reference modification — the foundation

Before the verbs, the most common string operation in any COBOL
program: **reference modification**. The notation `field(start:length)`
extracts a substring.

```cobol
01  WS-NAME    PIC X(30).
01  WS-INITIAL PIC X.

* WS-NAME = "MICHAEL                       "
MOVE WS-NAME(1:1) TO WS-INITIAL.    *> "M"
DISPLAY WS-NAME(8:7).               *> "       " — 7 spaces past the name
```

Three things to internalise:

1. **`start` is 1-based.** Position 1 is the first byte. There is no
   position 0.
2. **`length` is a length, not an end.** `WS-NAME(1:5)` is bytes 1
   through 5 (5 bytes), not "between 1 and 5."
3. **The length is optional.** `WS-NAME(8:)` means "from byte 8 to
   the end."

Reference modification works in any context that takes an
alphanumeric field — `MOVE`, `DISPLAY`, `IF`, the source of `STRING`,
the target of `UNSTRING`, etc. It does not work on packed-decimal or
zoned-numeric items unless you first redefine them as `PIC X`.

A real maintainer pattern, trimming a trailing-space-padded name to
just the meaningful prefix:

```cobol
* Find the last non-space position.
01  WS-NAME       PIC X(30).
01  WS-LAST-POS   PIC 9(3).

PERFORM VARYING WS-LAST-POS FROM 30 BY -1
        UNTIL WS-LAST-POS = 0
           OR WS-NAME(WS-LAST-POS:1) NOT = SPACE
END-PERFORM.

* WS-LAST-POS now holds the position of the last non-space, or 0.
DISPLAY "[" WS-NAME(1:WS-LAST-POS) "]".
```

This is the COBOL equivalent of `rtrim()`. Worth memorising.

## STRING — concatenation with delimiters

`STRING` builds a destination field by appending source operands. The
basic form:

```cobol
01  WS-FIRST   PIC X(15) VALUE "ALICE".
01  WS-LAST    PIC X(20) VALUE "WONDERLAND".
01  WS-FULL    PIC X(40).

STRING WS-FIRST  DELIMITED BY SPACE
       " "       DELIMITED BY SIZE
       WS-LAST   DELIMITED BY SPACE
    INTO WS-FULL
END-STRING.

DISPLAY "[" WS-FULL "]".  *> "[ALICE WONDERLAND                       ]"
```

The `DELIMITED BY` phrase decides how much of each operand gets
appended:

| Phrase                   | What it appends                                              |
| ------------------------ | ------------------------------------------------------------ |
| `DELIMITED BY SIZE`      | The operand's full declared length                           |
| `DELIMITED BY SPACE`     | Up to (but not including) the first space                    |
| `DELIMITED BY "literal"` | Up to (but not including) the first occurrence of the literal |
| `DELIMITED BY ws-field`  | Up to (but not including) the first occurrence of `ws-field`'s value |

For our example: `WS-FIRST DELIMITED BY SPACE` appends "ALICE"
(stops at the first space). `" " DELIMITED BY SIZE` appends a single
space. `WS-LAST DELIMITED BY SPACE` appends "WONDERLAND."

### POINTER — tracking position across multiple STRINGs

You can track the position in the destination across multiple STRING
operations:

```cobol
01  WS-FULL    PIC X(40).
01  WS-PTR     PIC 9(3) VALUE 1.

STRING WS-FIRST DELIMITED BY SPACE
    INTO WS-FULL WITH POINTER WS-PTR
END-STRING.

* WS-PTR now points just past the last byte written.

STRING " "      DELIMITED BY SIZE
       WS-LAST  DELIMITED BY SPACE
    INTO WS-FULL WITH POINTER WS-PTR
END-STRING.
```

The POINTER **must be initialised to 1** before the first STRING.
Forgetting this initialisation, or relying on COBOL's default (which
may be 0 depending on definition), is the canonical bug — the second
STRING starts at position 0 and corrupts.

### ON OVERFLOW — knowing when you ran out of room

`STRING` silently truncates if the destination runs out of room. The
`ON OVERFLOW` phrase catches it:

```cobol
STRING source-1 DELIMITED BY SIZE
       source-2 DELIMITED BY SIZE
    INTO WS-TARGET
    ON OVERFLOW
        DISPLAY "warning: STRING overflow at ptr=" WS-PTR
END-STRING.
```

The standard maintainer rule: **if the destination is smaller than
the worst-case sum of the sources, use `ON OVERFLOW`.** Without it
you have a silent-truncation bug waiting for the day a longer record
shows up.

## UNSTRING — parsing delimited input

`UNSTRING` is `STRING` in reverse. It splits a source field on a
delimiter and writes the pieces into a list of destination fields.

```cobol
01  WS-INPUT     PIC X(80) VALUE "ALICE,30,BOSTON,USA".
01  WS-NAME      PIC X(20).
01  WS-AGE       PIC X(3).
01  WS-CITY      PIC X(20).
01  WS-COUNTRY   PIC X(10).
01  WS-FIELD-CT  PIC 9(3).

UNSTRING WS-INPUT
    DELIMITED BY ","
    INTO WS-NAME
         WS-AGE
         WS-CITY
         WS-COUNTRY
    TALLYING IN WS-FIELD-CT
END-UNSTRING.

* WS-NAME = "ALICE", WS-AGE = "30", WS-CITY = "BOSTON",
* WS-COUNTRY = "USA", WS-FIELD-CT = 4.
```

Things worth remembering:

- **`TALLYING IN` gives you the actual field count.** Without it, you
  cannot distinguish "input had four fields" from "input had two
  fields and the last two destinations still hold their prior values."
  Always include it.
- **Each destination field gets the source piece padded or truncated
  to fit.** `WS-AGE` is 3 bytes; "30" gets two characters and a
  trailing space (or whatever the field's prior value held in byte 3
  if the field is left-justified).
- **DELIMITED BY ALL "x"** treats consecutive delimiters as one. Without
  `ALL`, two commas in a row mean the field between them is empty.
- **DELIMITER IN** captures which delimiter was found: useful if you
  have multiple delimiters in one UNSTRING (`DELIMITED BY "," OR ";"`).

## INSPECT — count, replace, tally

`INSPECT` walks a field looking for a pattern and either counts
matches, replaces them, or both.

### TALLYING — counting

```cobol
01  WS-LINE     PIC X(80) VALUE "name,age,city,country,zip".
01  WS-COMMAS   PIC 9(3) VALUE 0.

INSPECT WS-LINE TALLYING WS-COMMAS FOR ALL ",".

* WS-COMMAS = 4
```

Initialise `WS-COMMAS` to zero before INSPECT — TALLYING **adds** to
the counter, it does not reset.

### REPLACING — transforming

```cobol
INSPECT WS-LINE REPLACING ALL "," BY "|".

* "name|age|city|country|zip"
```

Common variations:

- `REPLACING ALL c1 BY c2` — replace every occurrence
- `REPLACING LEADING SPACES BY ZEROS` — typical for numeric padding
- `REPLACING FIRST c1 BY c2` — only the first occurrence
- `REPLACING CHARACTERS BY c2` — replace every character (the whole
  field becomes c2; rarely what you want)

### TALLYING + REPLACING combined

```cobol
INSPECT WS-LINE
    TALLYING WS-COMMAS FOR ALL ","
    REPLACING ALL "," BY "|".
```

One pass, two side effects. The compiler emits efficient code for
this; using two separate INSPECTs is wasteful.

### CONVERTING — character-class translation

```cobol
INSPECT WS-LINE CONVERTING
    "abcdefghijklmnopqrstuvwxyz"
    TO "ABCDEFGHIJKLMNOPQRSTUVWXYZ".
```

Lowercase to uppercase. CONVERTING is two parallel character lists
of equal length; each character in the first is replaced by the
corresponding character in the second.

## A worked example: parsing then rebuilding

A maintainer's pattern: read a comma-delimited record, validate the
field count, convert to upper case, and rewrite as pipe-delimited.

```cobol
01  WS-IN          PIC X(80).
01  WS-OUT         PIC X(120).
01  WS-FIELDS.
    05  F-NAME     PIC X(20).
    05  F-AGE      PIC X(3).
    05  F-CITY     PIC X(20).
    05  F-COUNTRY  PIC X(15).
01  WS-FIELD-CT    PIC 9(3) VALUE 0.
01  WS-PTR         PIC 9(3) VALUE 1.

PROCESS-LINE.
    UNSTRING WS-IN DELIMITED BY ","
        INTO F-NAME F-AGE F-CITY F-COUNTRY
        TALLYING IN WS-FIELD-CT
    END-UNSTRING.

    IF WS-FIELD-CT NOT = 4
        DISPLAY "bad record, " WS-FIELD-CT " fields, expected 4"
        EXIT PARAGRAPH
    END-IF.

    INSPECT F-NAME    CONVERTING "abcdefghijklmnopqrstuvwxyz"
                      TO "ABCDEFGHIJKLMNOPQRSTUVWXYZ".
    INSPECT F-CITY    CONVERTING "abcdefghijklmnopqrstuvwxyz"
                      TO "ABCDEFGHIJKLMNOPQRSTUVWXYZ".
    INSPECT F-COUNTRY CONVERTING "abcdefghijklmnopqrstuvwxyz"
                      TO "ABCDEFGHIJKLMNOPQRSTUVWXYZ".

    MOVE 1 TO WS-PTR.
    STRING F-NAME    DELIMITED BY SPACE
           "|"       DELIMITED BY SIZE
           F-AGE     DELIMITED BY SPACE
           "|"       DELIMITED BY SIZE
           F-CITY    DELIMITED BY SPACE
           "|"       DELIMITED BY SIZE
           F-COUNTRY DELIMITED BY SPACE
        INTO WS-OUT WITH POINTER WS-PTR
        ON OVERFLOW
            DISPLAY "output overflow"
    END-STRING.
```

Three patterns worth marking:

- `TALLYING IN WS-FIELD-CT` then a check — defensive parse.
- `INSPECT ... CONVERTING` for case fold — one pass per field.
- `STRING ... WITH POINTER ... ON OVERFLOW` — building output with
  position tracking and a defined failure path.

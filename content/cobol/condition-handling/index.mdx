# Condition handling

Most COBOL programs you'll inherit will assume nothing ever fails. The
file is always there, the record is always valid, the arithmetic
always fits. Production reality breaks every one of those assumptions.
Condition handling is the discipline of making those failures
**explicit** — turning every implicit "and if this fails, the program
ABENDs" into a deliberate "and if this fails, this is what we do."

This lesson covers the language features for explicit handling
(`AT END`, `INVALID KEY`, `ON SIZE ERROR`, `FILE STATUS`,
`DECLARATIVES`) and the maintenance patterns that put them to work.

## FILE STATUS — the discipline

Every file in COBOL can have a `FILE STATUS` variable declared in the
`SELECT` statement:

```cobol
ENVIRONMENT DIVISION.
INPUT-OUTPUT SECTION.
FILE-CONTROL.
    SELECT CUSTOMER-FILE
        ASSIGN TO CUSTFILE
        ORGANIZATION IS INDEXED
        ACCESS MODE IS SEQUENTIAL
        RECORD KEY IS CUST-ID
        FILE STATUS IS WS-CUST-STATUS.

DATA DIVISION.
WORKING-STORAGE SECTION.
01  WS-CUST-STATUS    PIC X(2).
```

After every I/O verb on `CUSTOMER-FILE` (`OPEN`, `CLOSE`, `READ`,
`WRITE`, `REWRITE`, `DELETE`, `START`), the runtime sets
`WS-CUST-STATUS` to a 2-character code describing what happened.

The codes are standardised:

| Status | Meaning                                                        |
| ------ | -------------------------------------------------------------- |
| `00`   | Successful                                                     |
| `02`   | Successful, but with a duplicate key found (informational)     |
| `04`   | Successful, but record length wrong (informational)            |
| `10`   | End of file (set by READ when the file exhausts)               |
| `21`   | Sequence error — keys out of order on a sequential write       |
| `22`   | Duplicate key on a write that doesn't allow it                 |
| `23`   | Record not found (READ KEY, START, REWRITE on missing key)    |
| `24`   | Boundary violation (write past defined record length)          |
| `30`   | Permanent I/O error (a real hardware/system problem)           |
| `34`   | Boundary violation (write past file's declared maximum)        |
| `35`   | OPEN input on a file that does not exist                       |
| `37`   | OPEN incompatible — wrong access mode for the file's RECFM     |
| `39`   | Conflicting OPEN (already open, or RECFM/ORG mismatch)         |
| `41`   | OPEN on a file that's already open                             |
| `42`   | CLOSE on a file that isn't open                                |
| `43`   | REWRITE/DELETE without a preceding successful READ             |
| `46`   | Sequential READ after a previous READ failed at end-of-file    |
| `47`   | OPEN INPUT on file that's been closed                          |
| `93`   | (z/OS) Resource not available, often VSAM in exclusive use     |
| `96`   | (z/OS) DD missing                                              |

The discipline: **every I/O operation is followed by a status check.**
The pattern most shops use is a paragraph called something like
`CHECK-CUST-STATUS`:

```cobol
READ CUSTOMER-FILE
    AT END SET END-OF-CUST-FILE TO TRUE
END-READ.
PERFORM CHECK-CUST-STATUS.

* ... later ...

CHECK-CUST-STATUS.
    IF WS-CUST-STATUS NOT = "00" AND
       WS-CUST-STATUS NOT = "10"
        DISPLAY "CUSTOMER-FILE I/O failed: status="
                WS-CUST-STATUS
        DISPLAY "  last operation aborted"
        MOVE 16 TO RETURN-CODE
        STOP RUN
    END-IF.
```

Three things this gets right:

- **Explicit allow-list of acceptable codes.** `00` for success, `10`
  for EOF (often expected). Anything else is an error.
- **Single failure path** that logs and exits with a non-zero return
  code so the JCL conditional logic can react.
- **Centralised** — every read of `CUSTOMER-FILE` calls one paragraph.
  When you change the error policy, you change one place.

The maintainer's instinct: when reading a paragraph that does
`READ FILE-X` without a status check immediately after, that is a
candidate for a defensive-handling enhancement. Most legacy code
checks status only on `OPEN` and `CLOSE` and trusts everything in
between, which is exactly how a "permanent I/O error" turns into a
silent partial run.

## AT END and INVALID KEY — the verb-level handlers

Every file verb that can fail has a phrase for catching the most
common failure:

| Verb       | Phrase            | Fires on                               |
| ---------- | ----------------- | -------------------------------------- |
| `READ` (sequential) | `AT END`     | EOF reached                            |
| `READ` (random)     | `INVALID KEY` | Key not found in the file              |
| `WRITE`             | `INVALID KEY` | Duplicate key, sequence, or boundary   |
| `REWRITE`           | `INVALID KEY` | Same                                   |
| `DELETE`            | `INVALID KEY` | Key not found                          |
| `START`             | `INVALID KEY` | No record matches the START condition  |

The phrase is optional. **Without it, an unhandled file condition
ABENDs the program.**

```cobol
* Without AT END — ABENDs at EOF (typically U4038)
READ CUSTOMER-FILE INTO WS-CUST-REC.

* With AT END — explicit, no ABEND
READ CUSTOMER-FILE INTO WS-CUST-REC
    AT END
        SET END-OF-CUST-FILE TO TRUE
END-READ.
```

The maintainer's rule: every `READ` has an `AT END`. Every keyed
operation has an `INVALID KEY`. Either you handle it explicitly, or
you accept that production will hand you a U4038 some week.

## DECLARATIVES — central error handlers

The `DECLARATIVES` section is a special area at the top of the
`PROCEDURE DIVISION` for catch-all handlers:

```cobol
PROCEDURE DIVISION.
DECLARATIVES.

CUSTOMER-FILE-ERROR SECTION.
    USE AFTER STANDARD ERROR PROCEDURE ON CUSTOMER-FILE.
HANDLE-CUST-ERROR.
    DISPLAY "I/O error on CUSTOMER-FILE: status="
            WS-CUST-STATUS.
    DISPLAY "  last record key: " CUST-ID.
    MOVE 16 TO RETURN-CODE.
    PERFORM CLOSE-EVERYTHING.
    STOP RUN.

ANY-FILE-ERROR SECTION.
    USE AFTER STANDARD ERROR PROCEDURE ON OUTPUT.
HANDLE-OUTPUT-ERROR.
    DISPLAY "I/O error on an output file".
    MOVE 16 TO RETURN-CODE.
    PERFORM CLOSE-EVERYTHING.
    STOP RUN.

END DECLARATIVES.

* Regular paragraphs follow.

MAIN-LINE.
    OPEN INPUT CUSTOMER-FILE.
    ...
```

The `USE AFTER STANDARD ERROR PROCEDURE` declarative is invoked
**after** any I/O error on the named file (or class — you can also
say `INPUT`, `OUTPUT`, `I-O`, or `EXTEND`) **if the failing verb
didn't handle the condition explicitly**.

Two things to know:

1. **Verb-level phrases (`AT END`, `INVALID KEY`) take precedence.**
   If the READ has `AT END`, the EOF condition fires that phrase, not
   the declarative.
2. **Declaratives are for catch-all handling.** Use them to centralise
   "log and abort" logic instead of repeating it after every check.

DECLARATIVES are common in production code for I/O. They're also used
for error conditions during sort/merge (`USE AFTER ERROR ON SORT-FILE`).
Less commonly, but legally, for reporting error events on display
output.

## ON SIZE ERROR — arithmetic recap

From the data-layout module: arithmetic operations can lose precision
or overflow. `ON SIZE ERROR` catches the overflow case:

```cobol
ADD 1 TO WS-COUNTER
    ON SIZE ERROR
        DISPLAY "counter overflow"
        MOVE 0 TO WS-COUNTER
END-ADD.
```

In the context of condition handling: this is the arithmetic
equivalent of `AT END` on a READ. It moves a potential ABEND (silent
truncation that ruins downstream calculations) into an explicit
recovery path.

The composite rule: **any operation that can fail has a corresponding
clause for explicit handling. If you don't write the clause, you
inherit whatever the runtime does — which is usually 'ABEND.'**

## Class tests — IS NUMERIC and friends

For data validation **before** arithmetic or file writes, COBOL
provides class tests:

```cobol
IF WS-PARAM-IN IS NUMERIC
    COMPUTE OT-MULT = FUNCTION NUMVAL(WS-PARAM-IN)
ELSE
    DISPLAY "bad parameter: [" WS-PARAM-IN "]"
    MOVE 1.50 TO OT-MULT
END-IF.
```

The class tests are:

- `IS NUMERIC` — every byte is a valid digit, with a valid sign
  nibble for signed fields. The "is this safe to ADD" test.
- `IS ALPHABETIC` — every byte is A-Z, a-z, or space.
- `IS ALPHABETIC-UPPER` — every byte is A-Z or space.
- `IS ALPHABETIC-LOWER` — every byte is a-z or space.

`IS NUMERIC` is the most useful, by far. Use it before any MOVE from
character data into a numeric field, and before any arithmetic on a
field whose origin you don't fully control.

`IS NUMERIC` does **not** mean "looks like a number to a human." A
field with `"1.5O"` (with the letter O) is **not** numeric. A field
with `"$100.00"` is not numeric. A field with `"  100"` (leading
spaces) is not numeric.

If you need to convert "human-style" numbers (with signs, decimal
points, leading spaces), use `FUNCTION NUMVAL`:

```cobol
COMPUTE WS-AMOUNT = FUNCTION NUMVAL("  -100.50").
*  WS-AMOUNT = -100.50
```

`NUMVAL` accepts leading sign, leading spaces, decimal point, and
trailing spaces. It throws an exception if the input doesn't look
like a number — so still test with `IS NUMERIC` (or a custom validator)
first if the input is genuinely uncontrolled.

## Composing the patterns: a defended file loop

Here is what a production-grade read loop looks like:

```cobol
PROCESS-CUSTOMER-FILE.
    OPEN INPUT CUSTOMER-FILE.
    PERFORM CHECK-CUST-STATUS.

    PERFORM UNTIL END-OF-CUST-FILE OR ABORT-REQUESTED
        READ CUSTOMER-FILE INTO WS-CUST-REC
            AT END
                SET END-OF-CUST-FILE TO TRUE
            NOT AT END
                PERFORM VALIDATE-CUST-REC
                IF CUST-IS-VALID
                    PERFORM PROCESS-CUST-REC
                ELSE
                    PERFORM LOG-BAD-CUST
                END-IF
        END-READ
        PERFORM CHECK-CUST-STATUS
    END-PERFORM.

    CLOSE CUSTOMER-FILE.
    PERFORM CHECK-CUST-STATUS.
```

What this code communicates to the next maintainer who reads it:

- File operations are explicit about success and failure. There is no
  silent path.
- Records are validated before processing. Bad records get logged,
  not ABEND'd over.
- Every status is checked. The program will hard-fail on a permanent
  I/O error, not run to a wrong total.
- The `ABORT-REQUESTED` flag (typically set by a declarative or a
  signal handler) lets the loop exit cleanly mid-file.

This is not legacy "clever" code. It's slightly verbose, very
predictable, and impossible to misread. It is also what production
operations folks expect from any program they have to support at 2am.

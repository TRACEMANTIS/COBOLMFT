# Reading a SYSUDUMP

The job ABENDed with `S0C7`. You opened SDSF, found the failed step,
and the JESYSMSG points at a `SYSUDUMP` dataset. You browse it. The
first thing you see is some 80-column lines of hexadecimal that look
like nothing at all. Welcome to the maintainer's daily reality.

This lesson is about turning that wall of hex into a single line of
COBOL source code. The skill is mostly mechanical — find the PSW,
subtract a base, look up an offset, read the listing — but it does not
get taught anywhere except on the job, which is why most maintainers
remain afraid of dumps for years longer than they need to.

## Three flavors of dump

When a program ABENDs, one of three dump formats lands in your spool.
Knowing which is which saves time:

| Dump          | Provided by | Best for                                              |
| ------------- | ----------- | ----------------------------------------------------- |
| `SYSUDUMP`    | OS          | Compact snapshot — the failing program's address spaces, plus key control blocks. The most common in batch JCL. |
| `SYSABEND`    | OS          | Larger version of `SYSUDUMP` — more storage included. Use when you need to see far-away storage. |
| `CEEDUMP`     | Language Environment | Formatted, source-line-aware dump for COBOL/PL/I. Easier to read than `SYSUDUMP` if your program is LE-enabled. |

In practice: most modern batch programs run under LE and produce both
a `CEEDUMP` and (if the JCL allocates one) a `SYSUDUMP`. **Always read
the `CEEDUMP` first** — it pre-formats the diagnostics into something
human-readable. Only fall back to `SYSUDUMP` if the `CEEDUMP` is missing
or you need raw storage to inspect.

The JCL DD statements look like:

```jcl
//SYSUDUMP DD  SYSOUT=*
//CEEDUMP  DD  SYSOUT=*
//SYSABEND DD  SYSOUT=*
```

If a dataset is allocated to one of these DDs, that is where the
formatted dump goes. Your shop's standard JCL almost certainly already
allocates them.

## The shape of a SYSUDUMP

A raw `SYSUDUMP` is roughly:

```
*** TITLE BLOCK ***       <- ABEND code, time, jobname
*** PSW AT ABEND ***      <- the program status word
*** REGISTERS ***         <- R0-R15 at the moment of failure
*** TRACEBACK ***         <- CALL chain (in newer dumps)
*** SAVE AREA TRACE ***   <- the linkage save areas, oldest to newest
*** STORAGE ***           <- pages of memory around the failing address
```

The same regions appear in a `CEEDUMP` with friendlier labels. The
order varies slightly by IBM release. The diagnostics process is
always the same: read the title, read the PSW, look up the failing
instruction in the listing, look at the failing field's bytes.

## The PSW — the failing instruction

The Program Status Word is a 64-bit (or 128-bit on z/Architecture)
register that holds the CPU's current state, including the address of
the **next instruction to execute**.

A typical SYSUDUMP entry:

```
PSW AT ENTRY TO ABEND  078D1000 004C8B78
```

The first 4 bytes (`078D1000`) are flags and mode; you can ignore them
for diagnostics. The second 4 bytes (`004C8B78`) are the instruction
address — the **next** instruction the CPU was about to run.

<Callout kind="warn" title="The PSW is one instruction past the failure">

A program-check exception (the `0Cx` family) is raised by the CPU
**after** it has decoded an instruction and decided it cannot complete
it. The PSW has already advanced to the next instruction's address.

To find the actual failing instruction, you back up: the previous
instruction is the failing one. Most COBOL-emitted instructions are 4
or 6 bytes long. So if PSW is `004C8B78`, the failing instruction is at
`004C8B72` or `004C8B74` (4 or 6 bytes earlier). The compile listing's
offset map will tell you the exact position.

</Callout>

## The compile listing — your decoder ring

When a COBOL program is compiled, the compiler can produce a listing
that includes the procedure-division offset map. This is opt-in via the
`LIST` and `OFFSET` (or `MAP`) compile options:

```
PP 5650-COB1 V6.4 IBM Enterprise COBOL for z/OS  CUSTBL01

PROCEDURE DIVISION REFERENCES BY VERB
SOURCE   COMPILER-GENERATED
LINE     OFFSET   VERB        OPERAND
   34    000A40   PERFORM     READ-CUSTOMER-FILE
   35    000A48   READ        CUSTOMER-FILE
   36    000A60   IF          CUSTOMER-EOF = 'Y'
   ...
   78    000B68   COMPUTE     RUNNING-TOTAL ROUNDED = ...
   78    000B70   ZAP         WS-WORKSPACE,WS-LINE-AMOUNT
   78    000B78   AP          WS-RUNNING-TOTAL,WS-WORKSPACE
   ...
```

The `OFFSET` column is the byte offset within the program's CSECT (the
load module's main entry). The `VERB` column tells you the
compiler-generated machine instruction. The `LINE` column is the COBOL
source line number.

**The mapping process:**

1. Read the PSW: e.g., `004C8B78`.
2. Find the program's CSECT base address. This is usually in a
   `LINKAGE EDITOR` map at the top of the dump, or in the
   `Module Map` section of a CEEDUMP. Suppose it's `004C8000`.
3. Subtract: `004C8B78 - 004C8000 = 0B78`.
4. The PSW points one instruction past the failure, so the failing
   offset is `0B70` or `0B72` (back up by 4 or 6 bytes).
5. Look up `0B70` in the compile listing. The line above shows
   `0B70 ZAP WS-WORKSPACE,WS-LINE-AMOUNT` on COBOL source line 78.
6. Source line 78 is `COMPUTE RUNNING-TOTAL ROUNDED = ...`.

That is the failing statement. Now you know **where** the program
faulted; you still need to figure out **why**.

## Reading the failing field's bytes

For an S0C7, the operands of the failing decimal instruction are the
two fields. The dump's storage section shows you their bytes.

```
STORAGE AT ENTRY TO ABEND
00510200   00000000 00000000 12345678 9C000000   *................*
00510210   40404040 40404040 00000000 00000000   *        ........*
00510220   00000000 00000000 20253103 00000000   *.........%......*
00510230   00000000 00000000 00000000 00000000   *................*
```

Each line is a 16-byte block; the address on the left is the start of
that block; the hex in the middle is the bytes; the EBCDIC translation
on the right (with dots for non-printable) is a sanity check. (`40` is
EBCDIC space.)

Looking up the operand addresses from the failing `ZAP`:

- `WS-LINE-AMOUNT` lives at `00510208` (per the listing or the symbol
  map). Its bytes are `12 34 56 78 9C` — packed `+12345678.9` — valid.
- `WS-WORKSPACE` is at `00510228`, 5 bytes: `20 25 31 03 00`.

That last byte is `00`. The low nibble is `0` — not a valid sign nibble
(`C`, `D`, or `F`). The `ZAP` instruction fails on the bad sign nibble
in the target.

Conclusion: `WS-WORKSPACE` is **uninitialized**. Some path through the
program reaches the COMPUTE without first setting `WS-WORKSPACE`. The
fix is either to `INITIALIZE WS-WORKSPACE TO ZERO` at the start of the
paragraph, or to find the path that skips the initialisation.

This is the canonical S0C7. Once you have done it three or four times
the muscle memory is permanent.

## The traceback — how did we get here?

The traceback (or "Save Area Trace") is the CALL chain. In a CEEDUMP
it is formatted readably:

```
Traceback:
   DSA Addr Program Unit  Entry      Statement   Load Mod    Service
   00510A00 CUSTBL01      CUSTBL01            78 CUSTBL01    LE6500
   005107A0 BATCHDR        BATCHDR            42 BATCHDR     LE6500
   00510700 CEEBINIT      CEEBINIT             0 CEEPLPKA    HLE7790
```

Read it bottom-up: `CEEBINIT` invoked `BATCHDR`'s entry point at
statement 42 (the `CALL`), which invoked `CUSTBL01`'s entry, which
ABENDed at statement 78. The "DSA Addr" is the dynamic save area for
each frame (used to find local variables; rarely needed for routine
diagnosis).

For a non-CALLing program the traceback is one entry — the failing
program itself. For a CICS or batch program with deep CALLs, the
traceback tells you who handed off responsibility, which is critical
when the actual failure is in a copybook or a shared CALLed module.

## A minimal worked dump

Suppose the SYSUDUMP shows:

```
*** TITLE BLOCK
JOBNAME    BATCH001
STEPNAME   STEP010
PROC STEP  RUNJOB
ABEND CODE S0C7

*** PSW AT ENTRY TO ABEND
078D1000 004C8B78  REASON=00000000

*** MODULE MAP
LOADMOD    CSECT    BASE       LENGTH
CUSTBL01   CUSTBL01 004C8000   00001500

*** STORAGE AT 00510200
00510200   00000000 00000000 12345678 9C404040   *...........{    *
00510210   40404040 40404040 00000000 00000000   *        ........*
00510220   00000000 00000000 20253103 00000000   *.........%......*
```

Procedure:

1. PSW addr = `004C8B78`. CSECT base = `004C8000`. Offset = `0B78`.
2. Listing shows `0B70 ZAP WS-WORKSPACE,WS-LINE-AMOUNT` (source line 78).
3. The failing instruction is the ZAP at offset `0B70`.
4. Operands: `WS-LINE-AMOUNT` at `0051020C` is `12 34 56 78 9C` — valid.
5. `WS-WORKSPACE` at `00510228` is `20 25 31 03 00` — last byte `00`,
   invalid sign nibble.
6. Read source line 78. Identify the path that reaches this line
   without initialising `WS-WORKSPACE`.
7. Fix: add `MOVE ZERO TO WS-WORKSPACE` or `INITIALIZE` at the start
   of the paragraph.

That is the entire diagnostic loop. It takes about three minutes once
you have done it ten times.

## Compile options that make dumps useful

Three options change the quality of the listing — and therefore the
dump's diagnostic value:

| Option        | What it does                                                   |
| ------------- | -------------------------------------------------------------- |
| `LIST`        | Generate a procedure-division-by-procedure-division listing   |
| `OFFSET`      | Include the offset of every COBOL verb in the listing         |
| `MAP`         | Include a map of every data item with its offset in the program's storage |
| `SOURCE`      | Print the source code in the listing (default on)             |

The maintainer's checklist for any program that **might** ever ABEND in
production: compile with `LIST`, `OFFSET`, `MAP`. The listing is
slightly larger; the diagnostic value is enormous. Without `OFFSET`,
locating a failing instruction by address is far harder.

If you inherit a program compiled without these options, you compile
it yourself — same source, same JCL, but with the better options — to
produce a fresh listing for diagnosis. The PSW addresses change between
compiles, but the source structure does not, so the new listing is
valid against the old dump as long as the procedure-division logic
matches.
